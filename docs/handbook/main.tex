This handbook is written as a working engineering reference for the MMS V2 installation in Tinkerers' Lab. It is intended for coordinators and maintainers who need to operate the system, diagnose failures, issue and revoke cards, deploy firmware, and hand the system forward without relying on the original implementation team.

\begin{controlbox}{Maintainer Note}
Update this handbook when production behaviour changes. In particular, record firmware releases, wiring changes, Firebase schema changes, security decisions, and recovery procedure changes before the next coordinator handover.
\end{controlbox}

\chapter*{Executive Summary}
\addcontentsline{toc}{chapter}{Executive Summary}

The Machine Management System Version 2 (MMS V2) is an IoT-based access control and usage tracking platform for the Tinkerers' Lab at IIT Indore. It controls access to five workshop machines (3D printer, laser cutter, grinder, two soldering stations) using RFID cards and logs all usage to a cloud database.

MMS V2 replaces a non-functional V1 system. The V1 system had a critical RFID block address mismatch, unhashed PIN storage, non-unique MQTT client IDs, no watchdog, no cloud logging, and was never tested end-to-end.

V2 is built on the ESP32 microcontroller, uses MIFARE Classic 1K cards with a custom cryptographic sector key scheme, routes session events through a Raspberry Pi MQTT broker to Firebase, and exposes a React web dashboard for real-time monitoring and administration.

This handbook serves as the complete reference for anyone deploying, maintaining, or extending MMS V2. It is written for longevity --- the assumption is that the original developers may no longer be available, and a future maintainer with general embedded systems knowledge should be able to understand and operate the system from this document alone.

\begin{controlbox}{Operational Promise}
MMS V2 is designed so machine access remains available during WiFi, Raspberry Pi, or Firebase outages. Safety-critical machine control remains local to the ESP32 node. Cloud services improve visibility and coordination; they are not required for the relay decision that starts or stops a session.
\end{controlbox}

%% --- Glossary and Acronyms ---
\chapter*{Glossary and Acronyms}
\addcontentsline{toc}{chapter}{Glossary and Acronyms}
\begin{longtable}{L{3cm}L{11cm}}
\toprule
\textbf{Term} & \textbf{Meaning} \\
\midrule
Access node & ESP32-based controller attached to one physical machine. \\
Bridge & Raspberry Pi service that moves MQTT events into Firebase and Firebase commands back to MQTT. \\
DESFire & More secure NXP card family recommended for future migration from MIFARE Classic. \\
FMEA & Failure Mode and Effects Analysis; a structured reliability review. \\
HC89 & Optical slot sensor used to detect whether an RFID card is physically inserted. \\
LittleFS & ESP32 flash filesystem used for cached revocation lists and offline session logs. \\
MFRC522 & RFID reader used by both access nodes and the card issuing station. \\
MQTT & Lightweight publish/subscribe protocol used between nodes and the Raspberry Pi. \\
OTA & Over-the-air firmware update delivered through Firebase Storage and MQTT command flow. \\
RTDB & Firebase Realtime Database, used for live node state and commands. \\
\bottomrule
\end{longtable}
